Terraform Fundamentals

What Does
TerraformActually Do?

No — Terraform is NOT just for writing infrastructure code. It declares, provisions, tracks, updates, and destroys real cloud infrastructure automatically.

🏗️ Infrastructure as Code ☁️ Multi-Cloud 🔄 State Management 🔌 Provider Ecosystem
Terraform does 4 things

It's an Infrastructure Automation Engine

01
DECLARE

You write HCL code describing what you want — a server, a database, a VPC. You don't write how to create it, just what the end result should look like.

02
PROVISION

Terraform calls the cloud provider's API (AWS, Azure, GCP, Tessell…) and actually creates the real infrastructure for you — VMs spin up, DBs are provisioned, DNS records get created.

03
TRACK

Terraform remembers everything it created in a state file (terraform.tfstate). This is how it knows what already exists vs what needs to change or be deleted.

04
SYNC

When you change your code, Terraform figures out the minimum set of changes needed to get from current state to desired state — no full rebuild, just a precise diff and apply.

Clarify

What Terraform Is vs What It Isn't

Many people think Terraform just "writes code". Here's the complete picture of what it actually covers.

📝
Language + Syntax
✅ Writes HCL config files ✅ Declares resources ✅ Variables + outputs ❌ Not a programming language

HCL (HashiCorp Config Language) is declarative — you say what, not how. No loops for provisioning logic, no imperative steps.

⚙️
Execution Engine
✅ Calls real cloud APIs ✅ Creates real servers/DBs ✅ Manages dependencies ❌ Not just "documentation"

When you run terraform apply, Terraform literally provisions real infrastructure. Money gets charged. Servers spin up. This is real, live automation.

🗃️
State Machine
✅ Tracks what exists ✅ Detects drift ✅ Plans minimal changes ❌ Not a deployment tool

Terraform's state engine is what makes it idempotent — run apply 10 times on the same config, it only changes what's actually different. No duplicate resources.

🧠 The Best Analogy

Terraform is like a blueprint + contractor + project manager all in one.

You write the blueprint (HCL code) → Terraform acts as the contractor who calls the right subcontractors (cloud APIs) to build what you described → Then Terraform keeps a ledger (state file) of exactly what was built, so future changes are surgical, not destructive.

Under The Hood

The Exact Execution Flow

Step by step — from your .tf file to a running database on Azure. Here's what actually happens at each stage.

📁

You Write .tf Files YOUR JOB

You write HCL files declaring what infrastructure you want. You describe the desired end state — database type, cloud region, compute size, network. Terraform doesn't care how to build it, just what the result should look like. These files are committed to Git like application code.

🔌

terraform init — Downloads the Provider COMMAND

Terraform reads your required_providers block and downloads the provider plugin from the Terraform Registry. For Tessell, it downloads tessell-cloud/tessell. This plugin is a binary that knows how to talk to Tessell's REST API. It gets cached in .terraform/. This only needs to run once per project setup.

🔍

terraform plan — The Diff Engine COMMAND

This is the most important step to understand. Terraform does 3 things:
1. Reads your .tf files — this is the desired state
2. Reads .tfstate — this is the known current state
3. Calls the cloud API to get actual current state (in case someone changed things manually)

Then it computes the exact diff — what needs to be created, changed, or destroyed to reach desired state. Nothing is created yet. This is pure calculation.

🚀

terraform apply — Real API Calls REAL INFRA

Now Terraform executes the plan. For each resource, it calls the provider plugin, which makes authenticated REST API calls to Tessell (or AWS, Azure, etc.). Tessell's API then provisions the actual Oracle VM, configures networking, sets up backup policies, and returns status. Terraform polls until the resource is READY. Real infrastructure is created. Real money is spent.

💾

State File Updated STATE

After every resource is created, Terraform writes the result to terraform.tfstate — a JSON file containing every resource's ID, attributes, and metadata. This is Terraform's memory. Next time you run plan, Terraform reads this file to know what already exists. If you delete this file, Terraform "forgets" about all the infrastructure it created and will try to create duplicates.

State Engine

How the State File Works

The state file is what makes Terraform intelligent. Without it, every apply would blindly recreate everything from scratch.

DESIRED STATE (.tf files)
engine_type "ORACLE"
compute_type "tesl_4_a"
region "eastUS"
topology "single_instance"
sla "PITR-7day"
→
DIFF
ACTUAL STATE (.tfstate)
engine_type "ORACLE" ✅
compute_type "tesl_4_a" ✅
region "eastUS" ✅
topology "single_instance" ✅
sla "PITR-1day" ⚠️
💡
Result of the diff above

Terraform sees only ONE difference — sla changed from PITR-1day to PITR-7day. So it only calls the Tessell API to update that single field. Everything else is untouched. The DB is NOT recreated. This is idempotency — surgical precision, not brute force.

Tool Approach What happens on re-run?
Bash scripts Imperative — step by step commands Creates duplicates or crashes
Ansible Procedural config management Mostly idempotent, but no real state
Terraform Declarative desired state Only changes what's different — always safe to re-run
Manual Console Click-ops No automation, no repeatability, no history

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 CONCEPT 01 — TERRAFORM BLOCK
 This is the root config block. It configures Terraform itself.
 Here you declare: which providers you need, minimum Terraform version,
 and where to store state (backend).
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
terraform {
  required_version = ">= 1.5.0"          # Minimum Terraform CLI version required

  required_providers {
    tessell = {
      source  = "tessell-cloud/tessell"  # Downloads from registry.terraform.io
      version = "~> 1.0"                # ~> means: 1.x but NOT 2.x (safe upgrades)
    }
    # You can have multiple providers — e.g., Azure for extra resources
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.0"
    }
  }

  # BACKEND: Where to store the state file
  # Default (no backend block) = local terraform.tfstate file
  # For teams, always use remote backend:
  backend "azurerm" {
    resource_group_name  = "terraform-state-rg"
    storage_account_name = "tfstatestorage"   # Azure Blob for shared state
    container_name       = "tfstate"
    key                  = "oracle-dev-qa.tfstate"
  }
}

 CONCEPT 02 — PROVIDERS
 Providers are plugins. They are the bridge between your HCL code and
 the cloud APIs. Each provider block configures one plugin.
 You can have multiple provider configurations for the same provider
 using the "alias" argument (e.g., two different Azure subscriptions).
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Provider 1: Tessell — manages Oracle DB provisioning
provider "tessell" {
  api_address = var.tessell_api_address
  tenant_id   = var.tessell_tenant_id
  api_key     = var.tessell_api_key
}

# Provider 2: Azure — for things outside Tessell (Key Vault, monitoring, etc.)
provider "azurerm" {
  features {}                              # Required empty block for azurerm
  subscription_id = var.azure_subscription_id
}


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 CONCEPT 03 — VARIABLES
 Variables are the inputs to your Terraform configuration.
 They let you reuse the same code for different environments (dev/prod).
 Types: string, number, bool, list, map, object, any
 Sources: terraform.tfvars, env vars (TF_VAR_xxx), -var flag, defaults
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# ── Simple string variable ──
variable "tessell_api_address" {
  type        = string
  description = "Tessell API base URL"
  default     = "https://api.tessell.com"  # Default = optional, no tfvars needed
}

# ── Sensitive variable (hidden in all output) ──
variable "tessell_api_key" {
  type      = string
  sensitive = true                        # Redacts from plan/apply/output logs
}

# ── Variable with validation ──
variable "environment" {
  type        = string
  description = "Target environment: dev, staging, or prod"
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "environment must be dev, staging, or prod."
  }
}

# ── Boolean variable ──
variable "enable_ha" {
  type    = bool
  default = false                          # false for dev, true for prod
}

# ── List variable ──
variable "allowed_ips" {
  type    = list(string)                  # List of strings
  default = []                            # Empty list by default
}

# ── Object variable (grouped config) ──
variable "db_config" {
  type = object({
    name     = string
    edition  = string
    compute  = string
    sla      = string
  })
  default = {
    name     = "oracle-dev"
    edition  = "STANDARD"
    compute  = "tesl_4_a"
    sla      = "PITR-1day"
  }
}

# ── Map variable (for per-environment config) ──
variable "env_settings" {
  type = map(object({
    compute        = string
    topology       = string
    backup_sla     = string
    deletion_prot  = bool
  }))
  default = {
    "dev" = {
      compute        = "tesl_4_a"
      topology       = "single_instance"
      backup_sla     = "PITR-1day"
      deletion_prot  = false
    },
    "staging" = {
      compute        = "tesl_8_a"
      topology       = "single_instance"
      backup_sla     = "PITR-3day"
      deletion_prot  = false
    },
    "prod" = {
      compute        = "tesl_16_a"
      topology       = "ha"
      backup_sla     = "PITR-7day"
      deletion_prot  = true
    }
  }
}

variable "vpc_name"                { type = string }

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 CONCEPT 04 — LOCALS
 Locals are computed values derived from variables or expressions.
 They are NOT inputs — they are internal computed constants.
 Use them to avoid repeating complex expressions throughout your code.
 Think of them like "const" variables in JavaScript.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
locals {
  # Pick settings based on current environment — like a switch statement
  current_env   = var.env_settings[var.environment]

  # Compute the DB name with environment suffix automatically
  db_name       = "${var.db_config.name}-${var.environment}"

  # Dynamic topology based on enable_ha flag
  topology      = var.enable_ha ? "ha" : "single_instance"

  # Merge standard tags with environment-specific tags
  common_tags = {
    "managed-by"  = "terraform"
    "environment" = var.environment
    "team"        = "engineering"
    "project"     = "tessell-oracle"
  }

  # Boolean: is this a production environment?
  is_production = var.environment == "prod"

  # Backup start time — earlier window in prod
  backup_time   = local.is_production ? "00:00" : "02:00"
}


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 CONCEPT 05 — DATA SOURCES
 Data sources READ existing infrastructure — they do NOT create anything.
 Use them to look up resources that already exist (created outside Terraform,
 or created by a different Terraform project).
 Syntax: data "provider_type" "local_name" {}
 Reference: data.provider_type.local_name.attribute
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Look up an existing Azure Resource Group (not managed by this Terraform)
data "azurerm_resource_group" "main" {
  name = "my-existing-resource-group"    # Must already exist in Azure
}

# Look up an existing Azure Key Vault secret for the DB password
data "azurerm_key_vault_secret" "db_password" {
  name         = "oracle-master-password"
  key_vault_id = "/subscriptions/.../resourceGroups/.../vaults/my-kv"
  # Now use: data.azurerm_key_vault_secret.db_password.value
  # This fetches the secret at apply time — no password in .tf files!
}

# Fetch current client's IP dynamically (no need to hardcode IP in tfvars)
data "http" "my_ip" {
  url = "https://ifconfig.me/ip"        # Returns your public IP as plain text
}


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 CONCEPT 06 — MAIN RESOURCE (Tessell Oracle DB)
 This is where Terraform actually CREATES infrastructure.
 Every argument here maps to a Tessell API field.
 Terraform sends these values to the Tessell REST API on apply.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
resource "tessell_db_service" "oracle_primary" {

  # ── Identity ─────────────────────────────────────────────────
  name         = local.db_name                # e.g. "oracle-dev-dev"
  description  = "Oracle DB | env: ${var.environment} | managed by Terraform"
  subscription = "default"

  # ── Engine ───────────────────────────────────────────────────
  engine_type            = "ORACLE"
  edition                = var.db_config.edition    # From object variable
  topology               = local.topology            # Computed from local
  software_image         = "Oracle 19c SE2"
  software_image_version = "19.17.0.0.221018"

  # ── Lifecycle Flags (env-aware) ──────────────────────────────
  auto_minor_version_update  = true
  enable_deletion_protection = local.current_env.deletion_prot  # true only in prod
  enable_stop_protection     = local.is_production               # conditional
  block_until_complete       = true

  # ── Infrastructure (Azure) ───────────────────────────────────
  infrastructure {
    cloud                  = "azure"
    region                 = var.azure_region
    availability_zone      = null
    vpc                    = var.vpc_name
    compute_type           = local.current_env.compute    # Different per env
    enable_encryption      = true
    encryption_key         = "default-encryption-key"
    additional_storage     = local.is_production ? 100 : 0 # 100GB extra in prod
    timezone               = "Asia/Calcutta"
    enable_compute_sharing = !local.is_production            # Share only in non-prod
  }

  # ── Network ──────────────────────────────────────────────────
  service_connectivity {
    service_port         = "1521"
    enable_public_access = !local.is_production            # Public only in dev/staging
    enable_s_s_l         = true
    allowed_ip_addresses = concat(
      var.allowed_ips,                                       # IPs from tfvars
      [trimspace(data.http.my_ip.response_body)]             # Your current IP (dynamic)
    )
  }

  # ── Database ─────────────────────────────────────────────────
  databases {
    database_name        = "PRIMARYDB"
    database_description = "Primary Oracle DB for ${var.environment}"
    database_configuration {
      oracle_config {}
    }
  }

  # ── Credentials ──────────────────────────────────────────────
  creds {
    master_user     = "admin"
    # Use Key Vault secret in prod, tfvars value in dev
    master_password = local.is_production
      ? data.azurerm_key_vault_secret.db_password.value
      : var.db_master_password
  }

  # ── Backup ───────────────────────────────────────────────────
  snapshot_configuration {
    auto_snapshot       = true
    sla                 = local.current_env.backup_sla      # Different SLA per env
    snapshot_start_time = local.backup_time                # Computed from local
  }

  # ── Dynamic Tags from map ────────────────────────────────────
  # dynamic block = loop inside a resource (like forEach in code)
  dynamic "tags" {
    for_each = local.common_tags          # Iterates over the tags map
    content {
      name  = tags.key                    # e.g. "environment"
      value = tags.value                  # e.g. "dev"
    }
  }

  # ── Lifecycle Rules ──────────────────────────────────────────
  # These control HOW Terraform manages changes to this resource
  lifecycle {
    # If any of these attributes change, Terraform updates in-place (not destroy+recreate)
    ignore_changes = [
      description,                         # Don't care if description drifts
    ]
    # Prevents accidental destruction of this resource in production
    prevent_destroy = false               # Set true for critical prod DBs

    # create_before_destroy: spin up replacement BEFORE destroying old one
    # Useful for zero-downtime changes. Not always supported by providers.
    create_before_destroy = false
  }

  # ── Explicit Dependency ──────────────────────────────────────
  # Tells Terraform: provision this DB only AFTER the Azure RG is confirmed
  depends_on = [data.azurerm_resource_group.main]
}

 CONCEPT 07 — LOOPS: for_each
 for_each lets you create MULTIPLE resources from a list or map.
 Instead of copy-pasting the same resource block 5 times,
 use for_each to loop over a collection.
 Each iteration creates a SEPARATE independent resource.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Create one additional DB service for each name in the list
# e.g., var.additional_db_names = ["reporting", "analytics"]
# → Creates: tessell_db_service.oracle_extra["reporting"]
# →          tessell_db_service.oracle_extra["analytics"]
resource "tessell_db_service" "oracle_extra" {
  for_each = toset(var.additional_db_names) # toset() deduplicates the list

  name         = "oracle-${each.key}-${var.environment}"
  description  = "Additional Oracle DB: ${each.key}"
  subscription = "default"
  engine_type  = "ORACLE"
  edition      = "STANDARD"
  topology     = "single_instance"
  software_image         = "Oracle 19c SE2"
  software_image_version = "19.17.0.0.221018"
  block_until_complete   = true

  infrastructure {
    cloud        = "azure"
    region       = var.azure_region
    vpc          = var.vpc_name
    compute_type = "tesl_4_a"
    enable_encryption     = true
    encryption_key        = "default-encryption-key"
    timezone              = "Asia/Calcutta"
    enable_compute_sharing= true
  }

  service_connectivity {
    service_port          = "1521"
    enable_public_access  = true
    enable_s_s_l          = true
    allowed_ip_addresses  = [trimspace(data.http.my_ip.response_body)]
  }

  databases {
    database_name         = upper(substr(each.key, 0, 8)) # Oracle SID max 8 chars
    database_description  = "DB for ${each.key} workload"
    database_configuration { oracle_config {} }
  }

  creds {
    master_user     = "admin"
    master_password = var.db_master_password
  }

  snapshot_configuration {
    auto_snapshot       = true
    sla                 = "PITR-1day"
    snapshot_start_time = "03:00"
  }
}

 CONCEPT 08 — OUTPUTS
 Outputs expose values after terraform apply completes.
 Uses:
   1. Print connection info to the terminal

The Big Picture

Terraform vs Everything Else

Where does Terraform fit in the toolchain? What does it replace vs what it works alongside?

Tool What it does Works with Terraform?
Terraform Provisions infrastructure (VMs, DBs, networks) and manages their lifecycle This IS Terraform
Ansible Configures software INSIDE servers (install packages, edit files, run services) Yes — Terraform creates the server, Ansible configures it
Docker / Kubernetes Runs application containers Yes — Terraform can provision the K8s cluster itself
GitHub Actions / Jenkins CI/CD pipeline automation Yes — run terraform apply inside a pipeline
Azure Portal (click-ops) Manual resource creation via browser UI Replaced by Terraform — not repeatable or trackable
Bash scripts + az CLI Scripted resource creation Mostly replaced — scripts have no state management or idempotency
Pulumi Infrastructure as Code using real programming languages (Python, TS) Alternative to Terraform, not used alongside it
