🔧 What Terraform Does

Terraform reads your .tf files, compares desired state with current state (.tfstate), then calls the Tessell provider plugin to make the API requests needed to reach the desired state.

🔌 What the Provider Does

The tessell-cloud/tessell provider is a plugin that knows how to translate your HCL resource blocks into Tessell REST API calls — you don't call the API directly.

📦 What .tfstate Is

After apply, Terraform writes a terraform.tfstate file — a JSON snapshot of every resource it created. This is how Terraform knows what already exists. Never delete it.

⚠️ Idempotency

If you run terraform apply twice on the same config, it won't create a duplicate DB — Terraform compares state first and only changes what's different.

01

provider.tf

The authentication and plugin setup — first file Terraform reads.

🔌
provider.tf
Tells Terraform which plugin to use and how to authenticate with Tessell
Required
provider.tf 
terraform {                              # ① Root terraform block
  required_providers {
    tessell = {                          # ② Local name you use in this project
      source = "tessell-cloud/tessell"  # ③ Registry: namespace/provider-name
    }
  }
}

provider "tessell" {                      # ④ Configure the tessell provider
  api_address = var.api_address         # ⑤ Tessell API base URL
  tenant_id   = var.tenant_id           # ⑥ Your org's unique ID
  api_key     = var.api_key             # ⑦ Your personal auth key
}
  • ①
    terraform {} is the meta-block that configures Terraform itself — not your resources. It's where you declare provider requirements, backend config, and required Terraform version.
  • ②
    tessell (the local name) is what you reference in your provider "tessell" block and in resources like tessell_db_service. It's just a local alias.
  • ③
    source = "tessell-cloud/tessell" tells Terraform to download the plugin from the public Terraform Registry at registry.terraform.io/providers/tessell-cloud/tessell during terraform init.
  • ④
    provider "tessell" {} configures the downloaded plugin — passes authentication credentials so every API call the provider makes is authenticated.
  • ⑤⑥⑦
    var.xxx references pull values from variables.tf + terraform.tfvars. Never hardcode credentials directly here — they'd be in your git history forever.
💡
Why separate provider config from resources?

The provider.tf / main.tf split is a convention, not enforced by Terraform. Terraform merges all .tf files in the directory. The split keeps auth config separate from resource definitions — cleaner and easier to reuse across projects by swapping just provider.tf.

02

variables.tf

Declares the shape of inputs — types, descriptions, and validation. No actual values here.

📐
variables.tf
Declares all input variables — types and descriptions only, no values
Config
variables.tf 
variable "api_address" {
  type        = string
  description = "Tessell API endpoint URL"
}

variable "tenant_id" {
  type        = string
  description = "Your Tessell Tenant UUID"
}

variable "api_key" {
  type        = string
  sensitive   = true          # ① Redacts value from all Terraform output
  description = "Tessell API Key"
}

variable "allowed_ip" {
  type        = string
  description = "Your machine IP to whitelist for DB access"

  validation {                    # ② Optional: enforce IPv4 format
    condition     = can(regex("^(\\d{1,3}\\.){3}\\d{1,3}$", var.allowed_ip))
    error_message = "Must be a valid IPv4 address."
  }
}

variable "db_master_password" {
  type        = string
  sensitive   = true
  description = "Oracle master user password"
}
VariableTypeSensitive?Why it exists
api_addressstringNo Tessell API base URL — can vary per tenant/environment
tenant_idstringNo Your org's unique UUID — namespaces all resources
api_keystring✅ Yes Personal auth token — marks as sensitive to hide in logs/output
allowed_ipstringNo Your IP for firewall whitelist — changes machine to machine
db_master_passwordstring✅ Yes Oracle admin password — sensitive to prevent exposure

① sensitive = true

This doesn't encrypt the value — it only prevents Terraform from printing it in plan, apply, and output. The value is still stored in plaintext in .tfstate. Always encrypt your state file in production.

② validation blocks

Optional but powerful — catches bad input before Terraform even calls the API. Use can() + regex() for format checks. Run during terraform plan so you fail fast, not after 10 min wait.

⚠ Secret
terraform.tfvars

Critical — Add to .gitignore immediately

Run this right now: echo "terraform.tfvars" >> .gitignore
Also add: *.tfstate, *.tfstate.backup, .terraform/
If you accidentally commit terraform.tfvars, rotate all your API keys immediately — git history is permanent.

04

main.tf — Top-Level Resource Block

The core file. Every top-level parameter in tessell_db_service explained.

🏗️
main.tf — Identity + Engine Block
What the DB is called, what it runs, and how it's deployed
Core
main.tf — section 1/6: identity + engine 
resource "tessell_db_service" "oracle_dev_qa" {
  #         ↑ resource type          ↑ local name (used in outputs/refs)

  # ── Identity ──────────────────────────────────────────
  name         = "oracle-dev-qa"         # ① Visible in Tessell Console
  description  = "Oracle 19c Dev/QA"     # ② Human-readable context
  subscription = "default"              # ③ Tessell billing subscription

  # ── Engine Config ─────────────────────────────────────
  engine_type            = "ORACLE"              # ④ Engine family
  edition                = "STANDARD"            # ⑤ Oracle edition
  topology               = "single_instance"    # ⑥ HA or no HA
  software_image         = "Oracle 19c SE2"     # ⑦ Tessell image name
  software_image_version = "19.17.0.0.221018"  # ⑧ Exact patch version

  # ── Lifecycle Settings ────────────────────────────────
  auto_minor_version_update  = true   # ⑨ Auto-apply minor patches
  enable_deletion_protection = false  # ⑩ Allow destroy in Dev/QA
  enable_stop_protection     = false  # ⑪ Allow stop in Dev/QA
  block_until_complete       = true   # ⑫ Wait for READY before finishing
ParamValueExplanation
name"oracle-dev-qa" Must be unique within your Tessell subscription. Used as identifier in the console and for DNS endpoint naming.
subscription"default" Tessell billing + quota context. Check your Tessell console for exact subscription names — case sensitive.
engine_type"ORACLE" Supported values: ORACLE, POSTGRESQL, MYSQL, SQLSERVER, MONGODB. Determines available editions, images, and config options.
edition"STANDARD" Oracle Standard Edition 2 (SE2) — up to 2 sockets, sufficient for Dev/QA. Use ENTERPRISE for partitioning, RAC, Advanced Security, etc.
topology"single_instance" Single VM, no standby replica. For Dev/QA this is fine. In production use "ha" — creates primary + standby with automatic failover.
software_image_version"19.17.0.0.221018" Full Oracle patch set version. 19.17 = 19c RU 17, .221018 = October 2022 patch. Find available versions in Tessell Console → Software Images.
block_until_completetrue Very important. Without this, Terraform returns "success" the moment Tessell accepts the request — but the DB is still provisioning. Set true to make Terraform wait for the DB to actually be READY (~15–20 min).
05

infrastructure {} Block

Where the DB lives — cloud, region, compute shape, network, encryption.

☁️
main.tf — infrastructure {} block
Defines Azure region, compute shape, VPC, encryption, and timezone
Required
main.tf — section 2/6: infrastructure 
  infrastructure {
    cloud             = "azure"           # ① Cloud provider
    region            = "eastUS"          # ② Azure region
    availability_zone = null             # ③ null = Tessell picks AZ
    vpc               = "your-vpc-name"  # ④ Tessell VNet (exact name from console)

    compute_type      = "tesl_4_a"       # ⑤ Tessell compute shape
    enable_encryption = true            # ⑥ Encrypt data at rest
    encryption_key    = "default-encryption-key" # ⑦ Use Tessell-managed key
    additional_storage = 0             # ⑧ Extra GB on top of default

    timezone                = "Asia/Calcutta" # ⑨ Oracle system timezone
    enable_compute_sharing  = true           # ⑩ Share VM across services (cost saving)
  }
☁️
Cloud + Region
azure / eastUS
Tessell provisions the Oracle VM inside this Azure region. Other options: westEurope, centralIndia, southeastAsia.
🖥️
Compute Shape
tesl_4_a
4 vCPU Tessell shape. Maps to an Azure VM size internally. Check Tessell Console → Compute for all available shapes and their vCPU/RAM specs.
🔒
Encryption
true / default key
Encrypts data at rest using Tessell-managed keys. For production, use your own Azure Key Vault key instead of "default-encryption-key".
🌐
VPC / VNet
your-vpc-name
The Tessell-managed VNet where the DB is placed. Get the exact name from Tessell Console → Infrastructure → Networks. Case-sensitive.
🕐
Timezone
Asia/Calcutta
Sets Oracle's DBTIMEZONE and system clock. Critical for scheduled jobs, backup windows, and timestamp interpretation. Use IANA timezone names.
💰
Compute Sharing
true (Dev/QA only)
Multiple Tessell DB services share the same underlying VM. Reduces cost significantly in Dev/QA. Always set false in production for isolation.
06

service_connectivity {} Block

Controls how and who can reach the database over the network.

🌐
main.tf — service_connectivity {} block
Port, SSL, public access, and IP whitelist configuration
Required
main.tf — section 3/6: service_connectivity 
  service_connectivity {
    service_port         = "1521"          # ① Oracle default listener port
    enable_public_access = true           # ② Expose via public endpoint
    enable_s_s_l         = true           # ③ TLS in transit

    allowed_ip_addresses = [             # ④ IP whitelist (firewall rules)
      var.allowed_ip                      #    Only your machine's IP
    ]
  }

① service_port = "1521"

Oracle Net Listener default port. If you need non-standard port (e.g., 2484 for TCPS/SSL), change this. Your JDBC string must match: jdbc:oracle:thin:@host:1521:SID

② enable_public_access = true

OK for Dev/QA where you need direct access from your laptop. Never use true in production — use private endpoint + VPN/bastion instead. Public = internet-reachable endpoint.

③ enable_s_s_l = true

Encrypts all data in transit using TLS. Always enable this — even in Dev/QA. Prevents credential sniffing on the wire. Your JDBC driver must trust Tessell's cert or you need to add it to your truststore.

④ allowed_ip_addresses

Acts as a network firewall. Tessell creates Azure NSG (Network Security Group) rules behind the scenes. You can add multiple IPs or CIDR blocks: ["1.2.3.4", "10.0.0.0/24"]

✅
Pro tip: Dynamic IP lookup in Terraform

Instead of hardcoding your IP in tfvars, you can use a data source: add an HTTP data source to fetch ifconfig.me and use that value. This way terraform apply always whitelists your current IP automatically — useful if you're on a dynamic IP.

07

databases {} Block

Defines the logical Oracle database inside the service — the SID/PDB, character set, and engine config.

🗄️
main.tf — databases {} block
The Oracle database definition — name (SID), description, engine-specific params
Required
main.tf — section 4/6: databases 
  databases {
    database_name        = "DEVDB"                   # ① The Oracle SID/DB name
    database_description = "Dev/QA Oracle Database"  # ② Human label

    database_configuration {
      oracle_config {
        # Optional Oracle-specific parameters:
        # character_set        = "AL32UTF8"     ← Unicode (recommended)
        # national_character_set = "AL16UTF16"  ← NCHAR/NVARCHAR char set
        # parameter_profile_id = "xyz"          ← Custom init.ora params
      }
    }
  }
💡
DB Service vs Database — What's the difference?

DB Service = the infrastructure layer (VM, network, compute, storage, backups). Think of it as the Oracle installation.

database {} = the logical Oracle database (the SID/CDB) that lives inside the service. One service can host multiple databases. For most Dev/QA cases, one database per service is fine.

In Oracle terms: service ≈ the Oracle Home + listener. database ≈ the individual CDB/SID.

ParamExampleNotes
database_name"DEVDB" This becomes the Oracle SID. Max 8 chars, uppercase, alphanumeric only. Used in JDBC: @host:1521:DEVDB
character_set"AL32UTF8" Always use AL32UTF8 for new databases — supports all Unicode characters. Cannot be changed after creation without full rebuild.
parameter_profile_idfrom console Points to a Tessell Parameter Profile (like RDS Parameter Groups) — a named set of Oracle init.ora parameters. Leave empty to use default profile.
08

creds {} + snapshot_configuration {} + tags {}

Authentication, backup policy, and resource tagging.

🔑
main.tf — creds, snapshot, tags blocks
Master credentials, backup SLA, and resource tags
Required
main.tf — section 5/6: creds + snapshot + tags 
  # ── Master Credentials ────────────────────────────────
  creds {
    master_user     = "admin"                  # ① Oracle DBA user (SYS-level)
    master_password = var.db_master_password   # ② From tfvars, sensitive
  }

  # ── Backup / Snapshot Policy ──────────────────────────
  snapshot_configuration {
    auto_snapshot       = true          # ③ Enable automated backups
    sla                 = "PITR-1day"  # ④ Point-in-time recovery window
    snapshot_start_time = "02:00"      # ⑤ Backup window start (24hr UTC)
  }

  # ── Resource Tags ─────────────────────────────────────
  tags { name = "environment"; value = "dev-qa" }    # ⑥
  tags { name = "managed-by";  value = "terraform" }
  tags { name = "team";        value = "engineering" }
}  # ← closes tessell_db_service resource

① master_user

This becomes the Oracle DBA account with SYS-equivalent privileges. You connect with this to create schemas, grant permissions, etc. Common names: admin, dbadmin, dba.

④ SLA = "PITR-1day"

Point-In-Time Recovery window of 1 day — means you can restore to any second within the last 24 hours. For Dev/QA this is enough. Production typically needs PITR-7day or longer. SLA names come from Tessell Console → Policies.

⑤ snapshot_start_time

Backup begins at 2:00 AM (24hr format). Choose off-peak hours in your timezone. Tessell takes an Oracle RMAN snapshot — no downtime, but slight I/O impact during backup window.

⑥ tags

Tags are key-value pairs stored in Tessell metadata — not Azure tags directly, but visible in the Tessell console for filtering, cost attribution, and access control. Always tag managed-by = terraform so your team knows not to manually modify this resource.

📤
outputs.tf
Exports resource attributes after apply — usable by other Terraform modules or CI/CD
Output
outputs.tf 
output "db_service_id" {
  description = "Unique UUID of the provisioned DB service"
  value       = tessell_db_service.oracle_dev_qa.id
}

output "db_service_status" {
  description = "Current status — should be READY after apply"
  value       = tessell_db_service.oracle_dev_qa.status
}

output "db_service_name" {
  description = "Name as registered in Tessell"
  value       = tessell_db_service.oracle_dev_qa.name
}

output "oracle_port" {
  description = "Port to connect to Oracle"
  value       = "1521"
}

How outputs work

After terraform apply, all output values are printed to terminal AND saved in .tfstate. Run terraform output db_service_id anytime to retrieve them. Reference them in scripts: $(terraform output -raw db_service_id)

Cross-module usage

If this Tessell module is called from a parent module, outputs are accessible as module.tessell_oracle.db_service_id. This is how you pass the DB ID to an app deployment module in the same Terraform project.

Every command, what it does internally, and when to use it.

CommandWhat It Does InternallyWhen to Run
terraform init Downloads tessell-cloud/tessell provider from registry, creates .terraform/ lock file. Must run after any provider change. Once per project setup, and whenever you add/change providers
terraform validate Checks HCL syntax and schema validity without calling any APIs. Catches typos, missing required fields, wrong types. Before every plan — fast sanity check
terraform plan Calls Tessell API to read current state, compares to desired state in .tf files, shows a diff of what will change. No side effects. Always before apply — especially in CI/CD for review gates
terraform apply Executes the plan — calls Tessell API to create/modify/delete resources. Writes results to terraform.tfstate. With block_until_complete=true, polls until DB is READY. When you're ready to actually provision
terraform output Reads and prints output values from .tfstate. Use -raw flag for scripting. After apply to get connection info
terraform destroy Calls Tessell API to delete all managed resources. With enable_deletion_protection=false, this works cleanly. Teardown Dev/QA to save costs
⚠️
Never manually delete resources from the Tessell Console if Terraform manages them

If you delete the DB manually via Tessell Console but .tfstate still thinks it exists, Terraform will be confused. Always use terraform destroy for Terraform-managed resources. If you accidentally do this, use terraform state rm to remove the stale entry from state.
